# ProtoScope 项目协作规范

## 基本原则

1. 全程使用简体中文沟通、分析、注释和提交说明。
2. 优先执行可验证的实际操作，不用空泛建议替代推进。
3. 遵循 KISS 与 YAGNI：代码保持直接、清晰、可读，不提前实现当前任务不需要的抽象。
4. 任何未经验证的结论都必须明确标注为未验证。
5. 不回滚、不覆盖用户已有改动；遇到无关工作区变更时，只处理当前任务范围。

## 项目目标

ProtoScope 是一个基于 ImGui + ImPlot 的串口波形示波器，
支持通过 Lua 脚本完成数据协议解析与处理，
并支持将外部功能模块以 Dock 插件形式接入主界面。

`D:\Code\Cpp\ElfStaticView` 集成到本工程时，默认视为插件接入任务，
优先通过插件适配层挂入 Dock，不直接把其内部实现耦合进主应用核心模块。

## 新任务启动流程

每次开始一个全新任务时，必须先执行以下检查：

```powershell
git status --short --branch
git rev-parse --verify HEAD
```

分支规则：

1. 如果仓库已有提交，必须基于当前 `HEAD` 创建任务分支。
2. 分支名按任务类型命名：
   - `task/<short-topic>`：普通任务或功能开发。
   - `fix/<short-topic>`：缺陷修复。
   - `chore/<short-topic>`：工程、构建、文档维护。
3. 如果仓库还没有任何提交，不能声称已经基于当前提交创建分支；应先说明这是初始化仓库状态，并规划初始化提交边界。
4. 创建分支前如发现未提交改动，必须判断是否属于当前任务；不属于当前任务的改动不得擅自纳入提交。

## 任务计划文件

每次任务开始时，必须在系统临时目录写入分步骤计划文件。

Windows 环境使用：

```powershell
$env:TEMP
```

计划文件命名建议：

```text
ProtoScope-task-plan-<yyyyMMdd-HHmmss>.md
```

计划文件至少包含：

```markdown
# ProtoScope Task Plan

## Task
<任务名称>

## Branch
<分支名或初始化状态说明>

## Steps
- [ ] 1. 检查仓库状态
- [ ] 2. 创建或确认任务分支
- [ ] 3. 实施修改
- [ ] 4. 运行验证
- [ ] 5. 规划并创建提交

## Verification
- Pending

## Commit
- Pending
```

执行要求：

1. 每完成一个步骤，必须立即回写计划文件，将对应步骤更新为 `[x]`。
2. 任务结束时，必须记录实际运行过的验证命令。
3. 如果创建了提交，必须记录提交 hash。
4. 如果因为仓库状态、测试失败或用户确认缺失而没有提交，必须在计划文件中写明原因。

## 提交规划与自动提交

每次任务开始时，需要预先规划提交粒度；任务完成并验证后，应自动创建符合任务边界的提交。

提交粒度规则：

1. 文档修改：使用一个独立提交，例如 `docs: update project workflow rules`。
2. 构建系统或目录结构修改：使用独立提交，例如 `build: add app skeleton`。
3. 功能代码修改：按可独立理解的行为边界拆分提交。
4. 测试补充：如果与功能改动紧密相关，可以同提交；如果是大范围测试整理，应单独提交。
5. 不允许把无关文件、历史暂存内容或用户未授权改动混入当前任务提交。

提交前必须执行：

```powershell
git diff --check
git status --short --branch
```

如果本次任务涉及 CMake 工程或 C++ 代码，至少执行：

```powershell
cmake -S . -B build -G "Ninja"
cmake --build build
```

如果仓库已配置测试目标，再执行：

```powershell
ctest --test-dir build --output-on-failure
```

后台执行测试命令建议设置最大超时时间 60 秒，除非当前任务明确需要更长时间。

## 构建系统与工具链

1. 使用 Modern CMake 管理工程。
2. 默认优先兼容 GCC / Clang，也要保证 Windows 下工具链可正常构建。
3. C++ 标准使用 C++20。
4. CMake 配置优先使用 target 级写法，避免全局污染。
5. 不在源码中硬编码本机绝对路径。
6. 第三方依赖必须有清晰边界，不把依赖细节扩散到业务模块。

CMake 编写要求：

1. 优先使用 `target_compile_features`、`target_compile_options`、`target_include_directories`、`target_link_libraries`。
2. 可执行程序、库、插件、测试目标职责清晰。
3. 公共编译选项集中管理，但不要过早抽象复杂 CMake 框架。
4. 新增 CMake 文件时，应保证从干净构建目录可以重新配置和构建。

目录职责：

1. `src/app`：应用启动编排、窗口生命周期、主循环与顶层工作区装配。
2. `src/serial`：串口枚举、连接、读写与设备状态管理。
3. `src/capture`：原始字节流采集、时间戳、缓冲、队列与帧投递。
4. `src/protocol`：协议数据模型、解析接口与内建协议适配边界。
5. `src/scripting`：Lua 虚拟机、脚本加载、热重载、桥接与脚本错误处理。
6. `src/plot`：通道数据组织、曲线缓存与 ImPlot 绘制适配。
7. `src/dock`：Dock 面板、工作区布局与各类工具面板 UI。
8. `src/plugin`：插件接口、注册、生命周期与宿主适配。
9. `plugins/elf_static_view`：`ElfStaticView` 的插件接入层与宿主侧桥接代码。
10. `assets/scripts`：默认 Lua 示例脚本与调试脚本。
11. `tests/fixtures`：协议样本、串口输入样本与脚本测试数据。

## 架构边界

1. 串口采集链路按“设备 I/O -> 原始采集 -> 协议解析/脚本处理 -> 波形通道 -> UI 展示”分层。
2. Lua 脚本层负责协议解释与数据转换，不直接操作底层串口句柄或 ImGui 绘制细节。
3. Dock 面板负责界面组织与交互，不承载底层采集或解析实现。
4. 插件接入优先通过明确接口与适配层完成，避免插件实现反向污染主应用核心模块。
5. `ElfStaticView` 集成默认视为插件接入任务，而不是主应用核心模块改造任务。

## 禁止事项

1. 不提前创建职责不清的 `src/core`。
2. 不把串口 I/O、协议解析、Lua 执行和绘图逻辑混在同一模块。
3. 不在第一阶段提前实现复杂的动态插件发现、跨进程通信或脚本沙箱。
4. 不把 `ElfStaticView` 源码直接散落并耦合到主应用主干模块中，应优先通过插件适配层接入。
5. 不为尚未落地的协议类型、采样后端或插件形态提前设计通用大框架。

## 代码质量要求

1. 代码优先可读性和可维护性，不做过度封装。
2. 核心流程、关键算法、重要边界条件必须写简体中文注释。
3. 命名应表达真实领域含义，避免无意义缩写。
4. 新增核心逻辑必须考虑可测试性。
5. 模块依赖方向应清晰：采集层不依赖 UI，脚本层不直接耦合具体 Dock 面板，插件层通过接口接入宿主。
6. 公共接口要稳定、简洁；内部实现可以按需要迭代，不提前暴露。

## 测试与验证

1. 新增核心逻辑时，应同步补充单元测试。
2. 修复缺陷时，应优先补充能复现缺陷的测试。
3. 测试数据放在 `tests/fixtures`，避免散落到源码目录。
4. 后台执行测试时建议设置最大超时时间 60 秒，避免任务卡死。
5. 若验证无法执行，必须说明具体命令、失败原因和剩余风险。
6. 图形界面相关改动至少保证可配置、可构建；如果当前阶段无法做自动化 UI 验证，需要明确说明人工验证边界。

## 危险操作确认

执行以下操作前必须获得用户明确确认：

1. 删除文件或目录。
2. 批量移动或批量重命名文件。
3. 修改系统环境变量、权限或全局配置。
4. 全局安装、卸载或升级包管理依赖。
5. 删除数据库、结构变更或批量数据更新。
6. 调用生产环境 API 或发送敏感数据。

确认格式：

```text
危险操作检测！

操作类型：<具体操作>
影响范围：<详细说明>
风险评估：<潜在后果>

请确认是否继续？需要明确回复“是”、“确认”或“继续”。
```
